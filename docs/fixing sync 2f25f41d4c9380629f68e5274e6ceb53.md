# fixing sync

Flow Overview

- The “Sync now” button in the manage panel just calls POST /cohorts/{id}/sync with your NEXT_PUBLIC_API_KEY header (apps/web/src/components/cohort-
actions.tsx:10-149). If the API key is missing or wrong you’ll immediately see a 401/403 in the browser console and the button shows the red “Sync
failed” message.
- The API route enqueues letterboxd.tasks.run_cohort_pipeline on Celery, stores the Celery task id in cohorts.current_task_id, and returns JSON (apps/api/
routers/cohorts.py:502-535). That task orchestrates refresh → scrape → stats → rankings → enrichment and always clears current_task_id at the end (src/
letterboxd_scraper/pipeline/tasks.py:27-211).
- Progress in the UI comes from polling /cohorts/{id}/scrape-status every 5 s (only while the “Manage Cohort” accordion is open) and surfacing rows from
scrape_runs / scrape_run_members (apps/web/src/components/scrape-progress-panel.tsx:1-125, API endpoint at apps/api/routers/cohorts.py:163-220). Those
rows are written by record_scrape_run, mark_member_started/finished, etc. in src/letterboxd_scraper/services/telemetry.py:15-106.

How to see what’s happening

- Use your browser devtools (or curl) to make sure the POST [http://localhost:8000/cohorts/](http://localhost:8000/cohorts/)<id>/sync call actually succeeds and returns a task_id. If it
fails with 401/403, add/set NEXT_PUBLIC_API_KEY in apps/web/.env.local and ensure there is a user row whose api_key_hash matches (apps/api/auth.py). You
can generate a hash with python - <<'PY' … PY.
- Tail the API and worker logs to see whether the request was received and whether the Celery worker picked up the task:
    
    docker compose -f docker-compose.dev.yml logs -f api celery-worker
    You should see letterboxd.tasks.run_cohort_pipeline start almost immediately; if the worker log is quiet the worker probably isn’t connected to Redis.
    
- Verify both the API container and worker point at the same broker/result backend (TASK_QUEUE_BROKER_URL/TASK_QUEUE_RESULT_BACKEND in docker-
compose.dev.yml). If necessary, run docker compose exec redis redis-cli -n 0 llen stats (queue name is stats) to see whether tasks are piling up.
- If the POST returned a task id but nothing appears in logs, inspect the task directly via the Celery CLI:
    
    docker compose exec celery-worker celery -A letterboxd_scraper.pipeline.tasks result <TASK_ID> --traceback
    Any exception thrown in the pipeline will show up there.
    
- Inspect the database for the bookkeeping tables the UI relies on:
    
    docker compose exec api psql "$DATABASE_URL" \
    -c "select id,current_task_id,label,updated_at from cohorts order by id;" \
    -c "select id,job_name,status,last_error,started_at from job_runs order by id desc limit 5;" \
    -c "select id,status,run_type,notes from scrape_runs order by started_at desc limit 3;"
    If you see a failed job_runs row, grab its id and check scrape_run_members or job_runs.last_error for the exact failure message.
    
- When you want to confirm the front-end is refreshing, open the “Manage Cohort” panel; the scraper widget only mounts (and starts polling /scrape-status)
once the accordion is open, so leaving it collapsed makes it look like “nothing is happening” even though the pipeline might be running. You can always
hit /cohorts/<id>/scrape-status in a separate terminal to watch the numbers move.

Following the steps above should make it clear whether the button call is being rejected (auth), the Celery worker isn’t live, the pipeline crashed (check
job_runs/Celery result), or it’s running but the UI is just not watching it. Natural next steps once you see the root cause are to fix the failing worker
(if it crashed), address whatever exception shows up in job_runs, or wire up Flower/alerts if you want a live dashboard of Celery tasks.

The worker is still hitting the same 403 because we only spoofed the user agent. For accounts whose “following” pages require you to be
logged in, Letterboxd also checks for your lb_session (and friends) cookie. Right now the containers are still sending an empty cookie header
(LETTERBOXD_SESSION_COOKIE="" in .env.docker), so the request looks unauthenticated even though the UA matches Firefox.

To fix it:

1. Copy your real cookie: In Firefox/Chrome DevTools, open the [https://letterboxd.com/jesseonplex/following/](https://letterboxd.com/jesseonplex/following/) request, go to Storage/Application → Cookies
and copy the relevant entries, e.g. lb_session=… (and lb_tid, __cf_bm if present). Paste that exact string into .env.docker:
    
    LETTERBOXD_SESSION_COOKIE="lb_session=abc123; lb_tid=xyz"
    
    Keep the existing SCRAPER_USER_AGENT.
    
2. Restart the backend so the new env vars load:
    
    docker compose -f docker-compose.dev.yml restart api celery-worker
    
    (You don’t need to touch Redis.)
    
3. Verify the containers picked up the values (optional):
    
    docker compose -f docker-compose.dev.yml exec api env | grep -E 'SCRAPER|LETTERBOXD'
    docker compose -f docker-compose.dev.yml exec celery-worker env | grep -E 'SCRAPER|LETTERBOXD'
    
4. Trigger “Sync now” again. With both the UA and the cookie mimicking your browser session, the follow-graph crawl should stop returning 403, and the
pipeline can proceed.

If the cookie expires, repeat step 1 to refresh it.

our scraper becomes essentially obsolete without letterboxd api, right?

- Pretty much. We can keep chasing it by spoofing UA/cookies or slowing down requests, but it’s a moving target. If Letterboxd steps up enforcement—
requiring live bot detection checks, rotating tokens, or full login flows—we’re stuck reverse‑engineering each change with no guarantees. Without an
official API or a stable contract, the whole pipeline lives at their mercy: one tweak on their end and we’re back to square one. This quick fix gets you
scraping again today, but long term you’d need either a permissions-based API, an internal data feed, or to accept that the scraper may break anytime they
tighten the screws.

Positioning Strategy

- Pitch a complementary feature: Emphasize that the app enhances Letterboxd communities (e.g., cohort-based watch clubs, collaborative planning, or

scraping or bulk exports.

branding respectfully.
